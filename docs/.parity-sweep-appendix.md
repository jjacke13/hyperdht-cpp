### HIGH

- **dhtrpc-tick/tick-4** — Persistent transition derives node id/port from the CLIENT socket's external port, not the SERVER socket's (JS rebuilds the sampler from PING_NAT replies)
  - JS .analysis/js/dht-rpc/index.js:801-875 (_updateNetworkState: natSampler=new NatSampler at 818, id=peer.id(natSampler.host,natSampler.port) at 831, this._nat=natSampler at 842), 916-963 (_checkIfFirewalled samples natSampler from PING_NAT replies whose `to` carries our server port at 929/940 and verifies natSampler.port===serverSocket port at 954)

It then computes the persistent id and replaces this._nat from THAT server-derived sampler. C++ keeps the single nat_sampler_ that (while ephemeral/firewalled) was filled from ordinary responses to client_socket_ requests — i.e. the CLIENT socket's external port. do_persistent_transition computes the id from nat_sampler_.port() (client external), and the firewall-probe port-preservation check only validates the client port; the server-port PING_NAT replies are dropped by nat_sampler_'s visited_ dedup because those relay nodes were already sampled.
  - → After a successful ephemeral→persistent transition the C++ node advertises an id/port that does not match the socket it transmits from, defeating the purpose of going persistent (peers won't store it, announces/lookups keyed on its id misbehave).
- **query/commit-1** — Commit phase never signals failure: empty-closest, all-timeout, and tokenless-only all report SUCCESS
  - JS dht-rpc/lib/query.js:211-248,392-393; hyperdht/index.js:266-300 (await query.finished())
  - C++ src/query.cpp:520-560; include/hyperdht/query.hpp:152 (commit_success_ declared, never read)
  - JS _endAfterCommit: (a) if the commit set is empty (no closestReplies) it destroy()s the query with 'Too few nodes responded'; (b) each commit promise settles, and if the LAST one settles with success===0 it destroy()s with the error; (c) autoCommit rejects for a reply with no token, so tokenless replies count as failed commits. immutablePut/mutablePut await query.finished(), so all three destroy() paths REJECT to the caller.   - JS .analysis/js/hyperdht/lib/connect.js:138-153,453-487,780-784

- **messages/messages-1** — LOOKUP reply length-prefixes each peer record; JS uses c.raw (no prefix)
  - JS .analysis/js/hyperdht/lib/messages.js:317,324-327 (rawPeers=c.array(c.raw)); .analysis/js/compact-encoding/raw.js:6-17 (raw: no length prefix)

- **messages/messages-2** — LOOKUP reply returns full announce-message bytes as peers; JS returns bare peer records
  - JS .analysis/js/hyperdht/lib/persistent.js:129 (record = encodeUnslab(m.peer, peer)); :36 (reply with lookupRawReply {peers: records})

- **router-announce/announce-1** — LOOKUP reply uses a non-JS wire format (per-record length prefix, no bump, full-announce elements) — JS lookup misdecodes every C++ node's records
  - JS .analysis/js/hyperdht/lib/persistent.js:36; .analysis/js/hyperdht/lib/messages.js:319-334; .analysis/js/hyperdht/index.js:520-534
  - C++ src/rpc_handlers.cpp:480-504; src/rpc_handlers.cpp:581

  - → JS clients cannot discover peers whose announces are held by C++ DHT nodes; lookup returns empty for those records. bump signalling is also lost.
- **router-announce/announce-2** — FIND_PEER served from the announce store returns the full announce message, not the m.peer record — JS mapFindPeer misdecodes
  - JS .analysis/js/hyperdht/lib/persistent.js:39-43; .analysis/js/hyperdht/lib/persistent.js:129; .analysis/js/hyperdht/index.js:538-551
  - C++ src/rpc_handlers.cpp:449-455; src/rpc_handlers.cpp:581

- **router-announce/announce-3** — C++ DHT nodes never relay third-party PEER_HANDSHAKE/PEER_HOLEPUNCH (router else-branch unimplemented) — silent black hole
  - JS .analysis/js/hyperdht/lib/router.js:128-171; .analysis/js/hyperdht/lib/router.js:213-219
  - C++ src/router.cpp:80-84; src/router.cpp:172-181
